## 執筆を進める作業環境について

今回のdbt本では、複数の著者が共同で一冊の本書く体制になります。そのため、それぞれ並行で作業を進めつつ、コンフリクトしない体制が必要でした。

そこで執筆環境については[Re:VIEW](https://reviewml.org/ja/)を採用し、原稿の管理はgithubで行っています。また、著者ごとにディレクトリを切って、執筆を進めていきました。

```
dbt-book-2026/
  ├── authors/                   # 著者ごとの執筆ディレクトリ
  │   ├── gun/
  │   │   └── sample-chapter.re
  │   ├── nakaken/
  │   │   └── sample-chapter.re
  │   ├── tadaken3/
  │   │   └── sample-chapter.re
  │   └── mukai/
  │       └── sample-chapter.re
  │
  ├── images/
  │   └── nakaken/               # 著者ごとの画像ディレクトリ
  │       ├── dbt_er.png
  │       └── demo_er.png
  │   └── tadaken3/
  │       ├── dbt_er.png
  │       └── demo_er.png
  │
  └──　README.md                # プロジェクトの概要
```

こうすることで、各自、使い慣れたエディタを使って執筆活動に入れますし、お互いの作業を気にせずに、執筆を進め、mergeしていくことができました。

ドヤ顔で語っていますが、この執筆環境については、DATUM STUDIOの[いわさん](https://x.com/daaataaaaa1592)がおこなってくれました。改めてこの場を借りて感謝いたします。ありがとうございます。

### Re:VIEWとは
Re:VIEWは電子書籍と紙書籍のための、簡潔かつ強力なデジタル出版ツールです。コンピュータ書等の技術書籍（紙・電子）の商業出版や同人出版に利用されています。

Re:VIEWフォーマットという独自のフォーマットを覚える必要はあるものの、コマンド一発でビルド（原稿から書籍ファイルの出力）ができ、非常に便利です。

ボクの場合は、[Skim](https://skim-app.sourceforge.io/)というPDFビューワーを併用しながら、原稿を確認しつつ、執筆をしていきました。

![](https://storage.googleapis.com/zenn-user-upload/5a4829a11c91-20260325.png)*実際に執筆をしている様子*

## Claude Codeを使ってのレビュー体制
書籍執筆にあたっては、Claude Codeと協力しながら、作業を進めました。Claude CodeのAgent Teamsを使って、レビューをする体制を作りながら、原稿の内容を煮詰めていきました。

Agent Teamsは以下の体制で構築しました。
- 凄腕編集者：全体の取りまとめて、原稿の方向性についてアドバイスする
- CTO：技術的な観点でレビューをする
- リサーチャー：原稿の中で紹介しているコマンドが実際に動くか検証をする[^1]
- 読みやすさレビュワー：文章の読みやすさをレビューする。同じ助詞が続いてないかなどの文章のリズムの確認
- 校閲ガール：誤字、脱字がないかを確認する

特に読みやすさに関しては、tmknomさんの以下の記事や「[読みやすい技術書を書く技術](https://booth.pm/ja/items/1564937)」を参考にしました。

https://zenn.dev/tmknom/articles/readable-github-cicd-book

### Skillsで読みやすさレビューの観点を明文化する

実際には以下のような形でClaude CodeのSkills（一部抜粋）を作成して、適宜Claude Codeによるレビューを挟みながら執筆を進めました。

```
---
name: readability-reviewer
description: 技術書の「読みやすさ」を多角的にレビュー。文章・コード・スクリーンショット・紙面の観点から、読者体験を向上させるための具体的なフィードバックを提供。Re:VIEW形式の原稿ファイル(.re)のレビューに特化。
---

# 読みやすさレビュースキル

このスキルは、技術書の原稿（Re:VIEW形式）を「読みやすさ」の観点から体系的にレビューし、最高の読書体験を実現するための具体的なフィードバックを提供します。

## このスキルを使用するタイミング

- Re:VIEW原稿（.reファイル）の読みやすさをチェックしたいとき
- 文章・コード・図表・紙面構成の品質向上が必要なとき
- 読者にストレスを与える表現や構成を発見・修正したいとき
- 執筆ルールへの準拠を確認したいとき

## 設計原則

本レビューは以下の設計原則に基づきます：

- **シンプルにする**: 短い文章・簡潔なコード・直感的な図表へと磨き上げる
- **ノイズを減らす**: 難しい表現は避け、読者を迷わせる要素は徹底的に取り除く
- **テンポを重視する**: 読者の集中力を削がないよう、読んで疲れない紙面にする

## レビュー観点

### 1. 文章の読みやすさ

#### 1-1. 読点ルール
- 一文につき読点「、」が2つ以上使われていないか
- 読点が多い場合、文を分割または構造を組み換えることで改善できるか

#### 1-2. 助詞の多用
- 同じ助詞（特に「の」「を」）が一文中に繰り返されていないか
- 助詞のバリエーションで文の構造が明確になっているか

#### 1-3. 文体の統一
- ですます調で統一されているか（箇条書きなど完全な文でない箇所は例外）
- 表記ゆれ（同じ意味の語が異なる表記で使われていないか）

#### 1-4. 表現の明確さ
- あいまいな表現・口語表現が使われていないか
- 専門用語は適切に説明されているか
- 対象読者に適した表現レベルか

### 2. コードの読みやすさ

#### 2-1. コードの完全性
- コードは完全な状態（写経して動く形）で掲載されているか
- 説明に必要な箇所だけでなく、全体像が把握できる形になっているか

#### 2-2. コメントによる視線誘導
- 着目すべきポイントにコメントで誘導されているか
- コメントは「何をしているか」ではなく「どこに注目すべきか」を示しているか（実装詳細の自明なコメントは避ける）

#### 2-3. 詳細説明での再掲
- 長いコードの詳細説明時、該当箇所が再掲されているか
- 「全体から詳細へ」という構成の一貫性が保たれているか

#### 2-4. ファイルレイアウトの明示
- 複数ファイルにまたがるコードの場合、ファイルレイアウトが明示されているか
- どのコードがどのファイルにあるか、読者が迷わないか

### 3. スクリーンショット・図表の読みやすさ

#### 3-1. 操作手順の網羅性
- 初登場の画面はスクリーンショットが掲載されているか
- 複数操作がある場合、操作順序が明確に示されているか

#### 3-2. ノイズの除去
- スクリーンショットに不要な要素（関係ない画面の端、切れ端など）が含まれていないか
- 適切にトリミングされているか

#### 3-3. 図表の効果
- 図表が本文の理解を助けているか
- Mermaid図は直感的で分かりやすいか

### 4. 構成・紙面の読みやすさ

#### 4-1. 「全体から詳細へ」の構成
- 概念の説明 → 全体コード → 詳細説明という順序が守られているか
- 読者が迷子にならない論理的な順序で説明されているか

#### 4-2. 段落の完結性
- 各段落が一つのテーマに集中しているか
- 段落の長さが適切か（長すぎると読む負担が増える）

#### 4-3. Re:VIEW記法の適切な使用
- 見出しレベル（=、==、===）が適切に使われているか
- コードブロック（//list）、画像（//image）、インラインコード（@<tt>{}）が正しく使われているか
- コラム（===[column]〜===[/column]）が適切に活用されているか

## チェックリスト（クイックリファレンス）

### 文章
- [ ] 読点は一文につき最大ひとつ
- [ ] 同じ助詞の繰り返しがない
- [ ] ですます調で統一（例外: 箇条書きなど）
- [ ] 表記ゆれがない
- [ ] あいまい・口語表現がない

### コード
- [ ] 完全な状態で掲載されている
- [ ] 着目ポイントにコメントで誘導
- [ ] 詳細説明時に該当箇所を再掲
- [ ] 複数ファイルはレイアウトを明示
- [ ] 「全体→詳細」の順序で説明

### 図表・スクリーンショット
- [ ] 初登場画面はスクリーンショット掲載
- [ ] 操作順序が明確
- [ ] 不要なノイズが除去されている
- [ ] 図表が本文の理解を助けている

### Re:VIEW記法
- [ ] 見出しレベルが適切
- [ ] コードブロック・画像・インラインコードが正しく使われている
- [ ] コラムが適切に活用されている

### 常に実施すること

- ポジティブなフィードバックと改善提案のバランスを取る
- 日本語技術書の文体慣習を尊重する
- Re:VIEW記法の正しい使用を考慮する
- 具体的な行番号と改善例を示す
- 「読者目線」で問題を指摘する（「読者が迷う」「読者の負担が増える」など）

### 決して実施しないこと

- 一部の観点のみでレビューを終了しない
- 抽象的な指摘だけで終わらない（必ず具体的な箇所と改善案を示す）
- 批判的・否定的なトーンを使用しない
- 読者目線を失わない
```

このようにレビュー観点を明文化してSkillsにすることで再現性を持たせています。

## まとめ
今回は技術書典20に向けた執筆の中で実践した、執筆環境とClaude Codeを使ってのレビュー体制についてご紹介しました。

- **Re:VIEW + GitHub** で複数著者の並行執筆を可能にした

- **Skills**でレビュー観点を明文化することで、再現性のある品質チェックの仕組みを作った
